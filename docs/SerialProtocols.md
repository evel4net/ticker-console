# Serial Communication Protocols

- Raspberry Pi Pico offers integrated hardware on the board for all the following protocols (see the [pinout](RaspberryPiPico_pinout.pdf))

## Universal Asynchronous Receiver Transmitter (UART)
It is the most used serial communication protocol. It uses two data lines:
- **Tx (Transmit)** for transmitting bits
- **Rx (Receiving)** for receiving bits

Unlike synchronous systems which rely on synchronizing everything on a single clock, UART communication is **asynchronous**.
This means there is no shared clock between systems/devices and no control over data. As a consequence, sent data begins with a 
start bit, low level, and ends with a stop bit, high level, which are added by the sender to help the receiver synchronize.
It is important to choose the transmission speed and size of data beforehand, else there is the risk of receiving garbage data.

UART is not the most efficient because:
- extra starting and ending bits
- complex hardware required for sending and receiving data
- speed has to be the same

## Serial Peripheral Interface (SPI)
It is an interface bus used for sending data between microcontrollers and small peripherals over short distances.

Unlike UART protocol, SPI is a **synchronous** protocol. It uses separate lines for sending and receiving data and another shared "clock" for synchronizing. 
The "clock" consists of an oscillating signal sent along with data to tell receiver when to read the next bit from the data line.
Because of this shared clock signal, the transmission speeds are independent to each system.


- clock = oscillating signal to tell receiver when you read data from the data line; rising or falling edge of the clock signal (specified in the datasheet)

### Components
- **CLK / SCK (Clock)**: signal generated by the Controller / Master (typically microcontroller) and received by the Peripheral / Slave
- **MOSI (Master Out - Slave In)**: data line for sending data from Master to Slave
- **MISO (Master In - Slave Out)**: data line for sending response data from peripheral back to controller, if response is needed and expected (controller needs to know the size of response to generate an appropriate number of clock cycles)
- **CS / CSS (Chip/Slave Select)**: signal for choosing the peripheral with which to communicate when there are multiple peripherals connected (high signal held when peripheral disconnected from SPI bus, then pulled low for activating it)

When working with multiple peripherals, they can use the same SCK signal and the same MOSI and MISO data lines, but each 
peripheral needs a different CS signal.

### Advantages & Disadvantages of SPI
| Advantages                        | Disadvantages                                                                              |
|-----------------------------------|--------------------------------------------------------------------------------------------|
| faster than asynchronous serial   | more signal lines, which require more wires                                                |
| receiver can be a simple hardware | communication rules must be set in advance, cannot send random amounts of data at any time |
| support for multiple peripherals  | peripherals cannot talk between them so controller must control all of their communications |
|                                   | different CS lines for each peripheral                                                     |

## Inter-Integrated Circuit (I2C)
It combines the best features of UART and SPI.
Like SPI, it is a synchronous protocol and uses two communication lines (connected to a positive supply using a pull-up resistor) between devices: 
- **SDA (Serial Data Line)**: shared line for sending and receiving data, used by both Master and Slave
- **SCL (Serial Clock Line)**: line for sending the shared clock signal controlled by Master for synchronization

I2C supports connections between one or more Masters with one or more Slaves.

### Transmission Process
The data is organized in frames of 8 bits and transmitted as follows:
- Master sends a **start condition** to all Slaves to signal a transmission will begin
- Master sends the **address frame** (7-10 bits) to all Slaves, which contains a unique identifier of the peripheral the Master wants to communicate with
- Slaves compare the received address frame with their own address and if address matches, it sends a confirmation bit back to Master, else nothing is sent
- address frame is followed by a **Read/Write bit** for specifying if Master is sending or receiving data
- data is then sent in consecutive **frames** of 8 bits long, each followed by an **ACK (acknowledge)** or **NACK (no-acknowledge) bit** sent by the receiver to confirm to sender that data was received successfully
- at the end, Master sends a **stop condition** to signal that transmission ends 

The address frame is required because I2C doesn't have a Slave Select line like SPI for choosing the peripheral you want to communicate with.

### Advantages & Disadvantages of I2C
| Advantages                                 | Disadvantages                        |
|--------------------------------------------|--------------------------------------|
| uses only two wires                        | slower than SPI                      |
| support for multiple masters and slaves    | size of data frame limited to 8 bits |
| simpler hardware than UART                 | more complicated hardware than SPI   |
| confirmation bits after each frame of data |                                      |
